# 토사 세분류 실제 경계·순서 반영 구현계획서

> Graph parent: [[지층_모델링_조사자료]]

## 1. 배경

현재 3D 지질 뷰어는 토사 대분류(`strata_group=soil`) 내부의 `soil_type` 값을 이용해 `매립토`, `매립 점토`, `퇴적점토`, `퇴적 사질토`, `풍화토` 등의 세분류 솔리드를 생성한다.

그러나 현재 구현은 세분류별 전체 두께를 시추공별로 합산한 뒤, 전체 토사 두께 안에서 세분류 두께 비율을 보간하여 정해진 표시 순서대로 쌓는 방식이다. 이 방식은 시추공 기둥의 세부 구간은 잘 보존하지만, 지층 솔리드에서는 실제 시추공의 세부 지층 경계와 상하 순서를 충분히 반영하지 못한다.

따라서 다음 단계에서는 토사 세분류를 단순 면적/두께 배분 레이어가 아니라, 시추공별 실제 세부 지층 구간에서 파생된 경계면 기반 솔리드로 생성해야 한다.

## 2. 현재 방식의 한계

### 2.1 현재 데이터 흐름

```text
시추공 strata
  -> strata_group=soil 구간만 추출
  -> soil_type별 총 두께 합산
  -> 세분류별 두께 IDW 보간
  -> 전체 토사 두께 안에서 비율 배분
  -> SOIL_DETAIL_ORDER 순서대로 누적 적층
  -> soil_detail:<세분류명> mesh 생성
```

### 2.2 문제점

- 같은 세분류가 한 시추공 안에서 여러 번 반복되는 경우, 반복 위치가 사라지고 총 두께만 남는다.
- `매립토 -> 퇴적점토 -> 퇴적 사질토 -> 퇴적자갈`처럼 실제 순서가 있는 데이터도 전역 표시 순서에 맞춰 재배치될 수 있다.
- 세분류 경계면이 실제 시추공의 `depth_top`, `depth_bottom`을 직접 통과하지 않는다.
- 수직 단면은 이미 생성된 mesh를 자르기 때문에, 솔리드 자체가 실제 경계를 반영하지 않으면 단면도 실제 경계를 반영할 수 없다.
- 세분류가 없는 시추공과 있는 시추공 사이의 연결 규칙이 불명확하여 얇은 가짜 레이어가 넓게 퍼질 수 있다.

## 3. 목표

### 3.1 기능 목표

- 토사 세분류 솔리드가 시추공별 실제 `depth_top`, `depth_bottom`, `soil_type` 순서를 기준으로 생성되도록 한다.
- 각 세분류 경계면은 가능한 한 해당 세분류가 관측된 시추공의 실제 상·하부 경계를 통과해야 한다.
- 세분류가 없는 곳에는 해당 세분류가 무리하게 생성되지 않도록 한다.
- 수직 단면 절단 시 세분류 경계가 시추공 기둥의 실제 구간과 최대한 일치해야 한다.
- 기존 대분류 지층 체계(`토사`, `풍화암`, `연암`, `보통암`, `경암`)와 암반 보간 방식은 유지한다.

### 3.2 비목표

- 토사 외 암반 계열을 세분류하는 기능은 이번 범위에 포함하지 않는다.
- 지질기술자가 수동으로 단면 correlation line을 편집하는 고급 기능은 후속 단계로 둔다.
- 실제 Boolean 연산으로 mesh를 영구 분할 저장하는 기능은 포함하지 않는다.

## 4. 설계 원칙

1. 대분류 토사 하부 경계는 기존 hard constraint 기준을 유지한다.
2. 세분류 경계는 토사층 내부에서만 계산한다.
3. 세분류 솔리드의 총 두께 합은 항상 대분류 토사 두께를 초과하지 않아야 한다.
4. 동일 시추공 위치에서는 세분류 경계가 원본 시추 데이터와 일치해야 한다.
5. 세분류가 관측되지 않은 영역에서는 자연스럽게 pinch-out 또는 부모 토사 레이어로 흡수되어야 한다.
6. 수직 단면은 별도 맵핑이 아니라 실제 생성된 세분류 솔리드를 절단한 결과를 보여준다.

## 5. 권장 구현 방식

### 5.1 핵심 방향

현행 `세분류별 총 두께 보간`을 `세분류 접촉면(contact surface) 보간`으로 변경한다.

```text
기존:
세분류별 총 두께 -> 비율 배분 -> 순서 적층

개선:
시추공별 세부 구간 -> 세부 경계 접촉점 생성 -> 층서 sequence 정규화
  -> 경계면 보간 -> 경계면 정렬/클램핑 -> 세분류 솔리드 생성
```

### 5.2 데이터 구조

Worker 내부에서 다음 중간 구조를 생성한다.

```ts
type SoilDetailSegment = {
  boreholeId: number
  x: number
  y: number
  elevation: number
  detail: string
  depthTop: number
  depthBottom: number
  topElevation: number
  bottomElevation: number
  sequenceIndex: number
}

type SoilDetailBoundary = {
  key: string
  upperDetail: string | null
  lowerDetail: string | null
  x: number
  y: number
  elevation: number
  boreholeId: number
}
```

`sequenceIndex`는 각 시추공 내부의 실제 위에서 아래 순서이다. 전역 지층명 순서가 아니라 시추공별 실측 순서를 먼저 보존한다.

### 5.3 입력 정규화

1. `strata_group === "soil"` 또는 `soil_type`이 토사 세분류로 인식되는 구간만 추출한다.
2. 같은 시추공에서 인접하고 같은 세분류인 구간은 병합한다.
3. 오탈자는 `normalizeSoilDetailName()`으로 정규화한다.
   - 예: `매립 지길` -> `매립 자갈`
   - 예: `퇴적모래` -> `퇴적 사질토`
4. 토사 세분류가 비어 있는 구간은 `토사` 또는 `미분류 토사`로 보존한다.

## 6. 세부 경계면 생성 알고리즘

### 6.1 시추공별 세분류 구간 배열 생성

각 시추공에서 토사 내부 구간을 표고 기준으로 변환한다.

```text
topElevation = borehole.elevation - depth_top
bottomElevation = borehole.elevation - depth_bottom
```

예:

```text
NBH-2
0.0~9.2   매립토
9.2~12.6  퇴적토
12.6~25.6 풍화토
25.6~26.4 연암
```

토사 내부 세분류는 다음처럼 contact point를 만든다.

```text
지표면/매립토 상부
매립토/퇴적토 경계
퇴적토/풍화토 경계
토사/암반 경계
```

### 6.2 경계 유형

세분류 솔리드 생성을 위해 다음 경계를 구분한다.

| 경계 | 설명 | 필수 여부 |
|---|---|---|
| soil_top | 지표면 또는 토사 상부 | 필수 |
| detail_top | 특정 세분류의 상부 경계 | 선택 |
| detail_bottom | 특정 세분류의 하부 경계 | 선택 |
| soil_bottom | 토사 대분류 하부 | 필수 |

세분류가 하나만 있는 시추공은 해당 세분류가 `soil_top~soil_bottom` 전체를 차지하는 것으로 본다.

### 6.3 경계면 보간

각 세분류별로 `topSurface(detail)`, `bottomSurface(detail)`를 보간한다.

권장 1차 구현은 기존 Worker의 격자 구조와 일관성을 위해 IDW 기반으로 시작한다.

- 입력점: 해당 세분류가 관측된 시추공의 상부/하부 표고
- 보간 대상: `gx`, `gy` 격자
- 거리 가중: 기존 `idwGrid(points, gx, gy, 2)` 재사용
- 관측점 hard snap: 시추공 위치에 가까운 격자는 원본 경계값으로 스냅

RBF/TPS 기반 경계면 보간은 2차 단계로 검토한다. 초기에는 현재 모델 안정성을 유지하는 것이 우선이다.

### 6.4 층서 순서 정렬

세분류 경계면은 보간 후 다음 조건을 만족하도록 정렬한다.

```text
soilTop >= detailTop >= detailBottom >= soilBottom
```

서로 교차하는 경우에는 다음 우선순위를 적용한다.

1. 시추공 hard snap 값
2. 세분류가 관측된 영역의 경계값
3. 부모 토사 두께 범위
4. 최소 두께 임계값

두께가 임계값보다 작으면 해당 격자에서는 세분류를 pinch-out 처리한다.

권장 임계값:

```text
SOIL_DETAIL_MIN_THICKNESS_M = 0.05
SOIL_DETAIL_PINCHOUT_THICKNESS_M = 0.10
```

## 7. 세분류 존재 영역 제어

### 7.1 필요성

관측점이 적은 세분류를 전체 모델 영역에 무제한 보간하면 실제로 없는 세분류가 넓게 퍼질 수 있다. 따라서 세분류별 presence mask가 필요하다.

### 7.2 Presence mask

각 세분류별로 존재 가능성을 계산한다.

```text
presence(detail, x, y) =
  관측 두께 기반 IDW 값
  + 인접 시추공 동일 세분류 영향
  - 미관측 또는 명시 부재 시추공 영향
```

1차 구현에서는 단순 기준을 사용한다.

- 해당 세분류 두께가 있는 시추공을 positive point로 둔다.
- 토사층은 있으나 해당 세분류가 없는 시추공을 weak negative point로 둔다.
- `presence < threshold`이면 해당 세분류 두께를 0으로 처리한다.

이 방식은 이미 존재하는 `soilPresence`와 유사한 개념으로 구현할 수 있다.

## 8. 실제 시추공 순서 보존 전략

### 8.1 전역 고정 순서의 문제

`SOIL_DETAIL_ORDER`만으로 모든 세분류를 쌓으면 특정 시추공에서 실제 순서가 다를 때 데이터가 왜곡될 수 있다.

### 8.2 권장 전략

1차 구현에서는 다음의 절충안을 사용한다.

- 시추공별 실제 순서를 `sequence signature`로 저장한다.
- 프로젝트 전체에서 가장 많이 나타나는 순서를 대표 순서로 선택한다.
- 같은 대표 순서에 속하는 세분류는 경계면 기반으로 생성한다.
- 대표 순서와 충돌하는 시추공은 진단 경고를 표시하고, 해당 위치는 시추공 컬럼을 우선 신뢰한다.

예:

```text
대표 순서:
매립토 -> 매립 점토/매립 사질토/매립 자갈 -> 퇴적점토/퇴적 사질토/퇴적자갈 -> 풍화토
```

2차 구현에서는 시추공 간 상관관계 그래프를 만들고, 단면 방향 또는 Delaunay edge별로 지역 순서를 적용한다.

## 9. 솔리드 생성 방식

### 9.1 기존 함수 재사용

기존 `buildLayerSolidGeometryData(topGrid, bottomGrid, ...)`를 재사용한다.

각 세분류에 대해 다음 pair를 만든다.

```text
[
  ["soil_detail:매립토", topGrid, bottomGrid, signedGrid],
  ["soil_detail:퇴적점토", topGrid, bottomGrid, signedGrid],
  ...
]
```

단, `topGrid`와 `bottomGrid`는 더 이상 비율 배분으로 산정하지 않고, 실제 접촉면 보간 결과를 사용한다.

### 9.2 기존 토사 솔리드와의 관계

- 세분류 솔리드가 생성된 영역에서는 기본 `soil` 솔리드를 숨긴다.
- 세분류가 불확실하거나 없는 영역은 기본 `soil` 솔리드를 유지할 수 있다.
- 이때 중복 표시를 막기 위해 `soil_detail_coverage_grid`를 별도 계산한다.

## 10. 수직 단면 표시

수직 단면은 별도 색상 맵핑으로 처리하지 않는다.

개선 후에도 수직 단면은 다음 방식으로 동작한다.

```text
세분류 솔리드 생성
  -> Three.js clipping plane 적용
  -> stencil cap 생성
  -> cap이 각 세분류 mesh의 색상/해치를 표시
```

따라서 단면 품질은 세분류 솔리드의 경계면 품질에 직접 의존한다.

## 11. UI 영향

기존 지층 표시 제어 구조는 유지한다.

- `토사`를 끄면 모든 토사 세분류도 함께 꺼진다.
- `매립토`, `퇴적토`는 그룹 항목으로 유지한다.
- `매립 점토`, `매립 사질토`, `매립 자갈`, `퇴적점토`, `퇴적 사질토`, `퇴적자갈`은 하위 항목으로 유지한다.
- 색상 사용자화는 `soil_detail:<세분류명>` 키를 그대로 사용한다.

추가 UI는 1차 구현 범위에서는 넣지 않는다. 다만 진단용으로 개발 모드에서 다음 값을 표시할 수 있다.

- 대표 세분류 순서
- 순서 충돌 시추공 수
- 세분류별 생성 격자 비율
- 세분류별 최소/최대 두께

### 12.2 신규 유틸 후보

복잡도를 낮추기 위해 Worker 내부 로직을 다음 파일로 분리하는 방안을 권장한다.

```text
sites/viewer-3d/src/lib/soilDetailSurfaces.ts
```

후보 함수:

```ts
buildSoilDetailSegments(boreholes)
buildSoilDetailBoundaries(segments)
selectRepresentativeSoilSequence(segments)
buildSoilDetailPresenceGrids(segments, gx, gy)
buildSoilDetailBoundarySurfaces(segments, gx, gy)
buildSoilDetailLayerPairsFromSurfaces(...)
```

Worker 번들 제약 때문에 import가 어려우면 1차 구현은 `geoWorker.ts` 내부 함수로 두고, 추후 분리한다.

## 13. 단계별 구현 계획

### Phase 1. 진단 및 데이터 구조화

- 현재 `boreholes`에서 토사 세분류 구간을 `SoilDetailSegment`로 변환한다.
- 시추공별 세분류 순서 signature를 산출한다.
- 부산역 프로젝트 기준 대표 순서와 충돌 시추공 목록을 콘솔 진단으로 출력한다.

완료 기준:

- 각 시추공의 토사 세분류 구간 수, 총 토사 두께, 대표 순서가 확인된다.
- NBH-2 등 편집 데이터가 원본 `depth_top/depth_bottom` 기준으로 들어오는지 확인된다.

### Phase 2. 경계면 기반 세분류 layer pair 생성

- 기존 `buildSoilDetailThicknessGrids()`와 `buildSoilDetailLayerPairs()`를 대체할 신규 함수를 작성한다.
- 세분류별 상·하부 경계면을 IDW로 보간한다.
- `soilTop`, `soilBottom` 범위 안으로 클램핑한다.

완료 기준:

- 시추공 위치에서 세분류 경계가 원본 깊이와 허용오차 내에서 일치한다.
- 세분류 총 두께가 토사 대분류 두께를 초과하지 않는다.

### Phase 3. Presence mask와 pinch-out 적용

- 세분류별 positive/negative anchor를 만든다.
- 존재 가능성이 낮은 영역에서는 세분류 두께를 0으로 처리한다.
- 너무 얇은 세분류는 pinch-out 처리한다.

완료 기준:

- 특정 시추공 주변에만 있는 세분류가 전체 영역으로 과도하게 번지지 않는다.
- 세분류가 사라지는 경계에서 심한 수직벽이 줄어든다.

### Phase 4. 수직 단면 검증

- 수직 단면을 생성해 시추공 컬럼과 세분류 솔리드 경계를 비교한다.
- `section-cap-face`가 세분류 mesh 색상과 해치를 정확히 따르는지 확인한다.
- 수직과장 변경 시에도 단면 경계가 같은 실제 표고를 유지하는지 확인한다.

완료 기준:

- 시추공 근처 단면에서 세분류 경계가 시추공 기둥과 육안상 일치한다.
- `토사`, `매립토`, `퇴적토` 토글과 하위 세분류 토글이 솔리드와 컬럼에 동일하게 적용된다.

### Phase 5. 회귀 테스트 및 문서화

- 부산역 프로젝트와 기존 샘플 프로젝트를 대상으로 비교 스크린샷을 작성한다.
- 생성 mesh 수, vertex 수, Worker 처리 시간을 기록한다.
- 기존 대분류 지층 품질이 훼손되지 않았는지 확인한다.

완료 기준:

- 빌드가 통과한다.
- 부산역 프로젝트에서 토사 세분류 솔리드가 실제 시추공 데이터의 경계/순서를 반영한다.
- 기존 암반 지층, 지하수위, 수직 단면 기능에 회귀가 없다.

## 14. 검증 기준

### 14.1 수치 검증

시추공 위치에서 다음 오차를 계산한다.

```text
abs(model_detail_top - observed_detail_top) <= 0.10 m
abs(model_detail_bottom - observed_detail_bottom) <= 0.10 m
```

격자 해상도와 보간 특성 때문에 정확히 0이 되기 어려우므로, 초기 허용오차는 0.10m로 둔다. hard snap 개선 후 0.02m까지 낮출 수 있다.

### 14.2 시각 검증

- 시추공 기둥 세분류 색상과 지층 솔리드 단면 색상이 대응해야 한다.
- 동일 시추공 주변에서 세분류 순서가 뒤집히지 않아야 한다.
- 세분류가 얇은 곳은 자연스럽게 사라져야 하며, 넓은 가짜 박판으로 남지 않아야 한다.
- 단면 cap에 빈 공간이나 중복 색상 오염이 없어야 한다.

### 14.3 성능 기준

초기 기준:

```text
시추공 300개 이하, 격자 248x248 기준 Worker 처리 시간 5초 이내 목표
```

세분류 mesh가 늘어나면 vertex 수와 draw call이 증가하므로, 다음 최적화를 검토한다.

- 실제 존재하는 세분류만 mesh 생성
- 두께가 0인 격자 제거
- 세분류별 presence bbox 밖 geometry 생략

## 15. 위험 요소와 대응

| 위험 | 영향 | 대응 |
|---|---|---|
| 시추공마다 세분류 순서가 다름 | 전역 sequence로는 모든 위치를 만족하기 어려움 | 1차 대표 순서 + 충돌 진단, 2차 지역 sequence |
| 관측점이 적은 세분류가 넓게 번짐 | 가짜 지층 생성 | presence mask와 negative anchor 적용 |
| 경계면 교차 | 지층 역전, cap 오염 | soilTop/soilBottom 클램핑과 최소 두께 필터 |
| mesh 수 증가 | 렌더링 성능 저하 | 존재 세분류만 생성, 빈 geometry 생략 |
| 기존 토사 솔리드와 중복 | 색상 겹침, 단면 혼탁 | 세분류 coverage grid로 기본 soil 표시 제어 |

## 16. 권장 우선순위

1. 부산역 프로젝트의 실제 세분류 구간을 기준으로 진단 로그를 먼저 만든다.
2. 시추공 위치 hard snap을 만족하는 경계면 기반 layer pair를 구현한다.
3. presence mask를 적용해 세분류 확산을 제한한다.
4. 수직 단면에서 시추공 컬럼과 세분류 솔리드 경계를 비교한다.
5. 이후 필요하면 RBF/TPS 또는 지역 sequence correlation으로 고도화한다.

## 17. 결론

실제 세부 지층 경계와 순서를 반영하려면 단면에서 색을 후처리로 맵핑하는 방식이 아니라, 토사 세분류 솔리드 자체를 시추공별 세부 경계면 기반으로 다시 생성해야 한다.

이번 개선의 핵심은 `soil_type별 총두께`가 아니라 `soil_type별 상·하부 접촉면`을 모델링 단위로 삼는 것이다. 이렇게 변경해야 수직 단면, 지층 표시 제어, 색상 사용자화, 시추공 기둥 표시가 모두 같은 지층 구조를 바라보게 된다.
